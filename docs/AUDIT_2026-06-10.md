## P0 — критично, чинить в первую очередь

### 1. Системная дыра: endpoints доверяют `user_id` из запроса вместо сессии

Самая серьёзная проблема всего проекта. Большой пласт endpoints либо вообще
не имеет `Depends(get_current_user)`, либо имеет, но не сверяет владение
ресурсом. Любой (в ряде случаев даже неаутентифицированный) пользователь
читает и изменяет чужие данные, подставив чужой `user_id` / угадав UUID.

Подтверждённые места:

| Endpoint | Файл:строка | Проблема |
|---|---|---|
| `GET /api/v1/leads` | `app.py:440` | `user_id: int = WEB_DEMO_USER_ID` без auth — чтение чужого CRM |
| `GET /api/v1/leads/export.csv` | `app.py:572` | массовая выгрузка чужих лидов |
| `PATCH /api/v1/leads/{id}` | `app.py:803` | auth есть, проверки владения лидом нет |
| `PATCH /api/v1/leads/bulk` | `app.py:1799` | нет auth вообще, bulk-изменение любых лидов |
| `POST /api/v1/leads/{id}/draft-email` | `app.py:1597` | `body.user_id` без проверки |
| `PUT /api/v1/leads/{id}/mark` | `app.py:1906` | `body.user_id` без auth |
| `GET /api/v1/searches/{id}` | `routes/search.py:285` | любой search по UUID без проверки владения |
| `GET /api/v1/searches/{id}/leads` | `routes/search.py:298` | лиды любого search |
| `GET /api/v1/searches/{id}/export.xlsx` | `app.py:682` | экспорт чужого search |
| `GET /api/v1/searches/preflight` | `routes/search.py:65` | `user_id` из query |
| `GET /api/v1/teams` | `routes/teams.py:76` | `user_id: int` без auth — просмотр чужих команд |
| `GET /api/v1/teams/{id}` | `routes/teams.py:107` | то же |
| `PATCH /api/v1/teams/{id}`, `PATCH .../members/{id}`, `POST .../invites` | `routes/teams.py:116,149,211` | `body.by_user_id` принимается на веру — действия от имени любого юзера, эскалация ролей |
| `GET /api/v1/teams/{id}/members-summary` | `app.py:1952` | «Owner-only» в docstring, проверки нет |
| `/api/v1/templates/*` (весь CRUD) | `routes/templates.py:35,66,91,115` | `user_id` параметром, без auth |
| `GET /api/v1/stats` | `app.py:2020` | статистика любого юзера |
| `GET /api/v1/team` | `app.py:2070` | глобальный список всех пользователей системы без auth |

**Что делать:** один общий рефакторинг — везде убрать `user_id`/`by_user_id`
из параметров запроса, брать identity только из `Depends(get_current_user)`,
а на каждом ресурсе проверять владение (user_id лида/search/template ==
current_user.id или членство в team). Просмотр CRM тиммейта владельцем —
только через проверенный `member_user_id` + проверку роли owner.
Покрыть это тестами «user A не видит данные user B» на каждый роут.

### 2. Слабые fallback-секреты

- `config.py:65` + `adapters/web_api/auth.py:61-71` — `AUTH_JWT_SECRET`
  по умолчанию пустая строка; HMAC-ключ хеширования session-токенов
  становится `SHA256("")` — детерминированный и общеизвестный. При забытой
  env-переменной подделка session-токенов реалистична.
- `core/services/secrets_vault.py:28-46` — при отсутствии `FERNET_KEY`
  ключ деривируется из захардкоженного seed
  (`convioo-dev-fallback-key-do-not-use-in-prod`). Все OAuth-токены
  интеграций в БД расшифровываются известным ключом.

**Что делать:** в проде (определять по `RAILWAY_*`/`ENV`) — fail-fast на
старте, если `AUTH_JWT_SECRET` или `FERNET_KEY` не заданы. Fallback оставить
только для локалки/тестов.

## P1 — высокий приоритет

### Безопасность

3. **SSRF через DNS rebinding в webhooks** — `collectors/website.py:68-100`,
   `routes/webhooks.py:65`. `assert_public_url()` резолвит DNS при создании
   webhook, а реальный запрос идёт позже — TTL=0 позволяет подменить адрес
   на внутренний. Резолвить IP непосредственно перед запросом и
   коннектиться к проверенному IP (или повторять проверку в момент доставки).
4. **OAuth state replay в multi-replica** — `core/services/oauth_state.py:35`.
   Карта использованных nonce — process-local; при нескольких репликах на
   Railway state можно переиграть через другую реплику. Перенести в Redis
   `SET NX EX` (или в Postgres, раз Redis опционален).
5. **Гонка при refresh OAuth-токенов** — `core/services/oauth_store.py:132-241`.
   Два конкурентных запроса оба видят истёкший токен, оба рефрешат; для
   провайдеров, не переиздающих refresh_token, второй ответ может затереть
   валидный. Нужен лок per-user (advisory lock в Postgres или Redis lock).
6. **Email header injection в sequence-рассылках** — `queue/worker.py:332-344`
   и отсутствие валидации subject в `routes/sequences.py`. `lead.name` /
   `lead.website` подставляются в subject без чистки CRLF; шаблон при
   создании не валидируется. Стрипать `\r\n` из всех подстановок и subject.
7. **Prompt injection из контента сайтов** — `analysis/prompts/system.py:212-213`.
   Спарсенный текст сайта (до 1200 символов) интерполируется в prompt без
   ограждения. Обернуть в явные разделители «untrusted data», добавить
   инструкцию игнорировать команды внутри блока.

### Надёжность

8. **Доставка webhooks глотает ошибки** — `core/services/webhooks.py:73-92,179-181`.
   Любое исключение логируется и поглощается, ретраев/экспоненциального
   backoff нет, статус доставки не отражает реальность. Добавить retry
   (3–5 попыток с backoff) и записывать фактический статус в delivery log.
9. **Stripe webhook: TOCTOU идемпотентности** — `routes/billing.py:304-314`.
   Конкурентные дубликаты события проходят проверку до вставки. Вставлять
   `StripeEvent` первым шагом и применять событие только если INSERT прошёл
   (`ON CONFLICT DO NOTHING` + проверка rowcount).
10. **Таймаут поиска оставляет статус `running`** — `pipeline/search.py:332-346`.
    При timeout уже записанные лиды остаются, а search не переводится в
    `failed`; пользователь видит вечный «running». Переводить статус в
    failed/partial в `finally`.
11. **`scoring.py:95`** — `msg.content[0].text` упадёт IndexError на пустом
    `content` от Anthropic API. Проверять длину, fallback на эвристику.
12. **`google_places.py:333`** — `resp.json()` без проверки content-type:
    HTML-страница 502 от прокси кладёт enrichment с JSONDecodeError.
13. **N+1 в members-summary** — `app.py:1979-2016`: по 2 запроса на каждого
    члена команды. Переписать на 2 агрегирующих запроса с GROUP BY.

## P2 — средний приоритет

14. **Rate limiting только на auth-роутах** (`utils/rate_limit.py` используется
    лишь в `routes/auth.py`). Мутации team/invites/sequences/webhooks не
    ограничены. Добавить лимиты на дорогие и state-changing endpoints.
15. **Подпись исходящих webhooks без timestamp** — `core/services/webhooks.py:111-120`.
    Подписывается только тело; перехваченную доставку можно реиграть. Включить
    timestamp в подписываемую строку (как Stripe: `t=...,v1=...`).
16. **Yelp/Foursquare без пагинации** — `collectors/yelp.py:66`,
    `foursquare.py:59`: жёсткий потолок 50 результатов на источник.
17. **Hunter.io без circuit breaker** — `core/services/email_finder.py:29`:
    при недоступности API каждый лид ждёт полный timeout последовательно.
18. **Nominatim: User-Agent не на каждом запросе** — `collectors/osm.py:162`;
    риск бана по политике OSM.
19. **Сессии: фикс 30 дней без ротации** — `adapters/web_api/auth.py:183,193`.
    Украденный токен живёт весь срок. Sliding expiry или ротация токена.
20. **Reply-tracker: naive datetime** — `core/services/email_reply_tracker.py:141-143`:
    `.timestamp()` на потенциально naive `email_reply_last_checked_at`
    сдвигает окно сканирования. Принудительно `replace(tzinfo=UTC)`.
21. **Worker молча коннектится к `redis://localhost`** — `queue/worker.py:466`
    при отсутствующем `REDIS_URL`. Для воркера Redis обязателен — fail-fast.
22. **Frontend: `RequireAuth` глотает ошибки профиля** —
    `components/RequireAuth.tsx:35-45`: при 401 пользователь остаётся на
    странице со «сломанным» состоянием. Редиректить на /login.
23. **Frontend: `dangerouslySetInnerHTML` для переводов** —
    `app/developers/page.tsx:352`. Сейчас данные статичны, но это ружьё на
    стене — заменить на обычный JSX.
24. **Mobile: страница поиска** — `app/app/search/page.tsx:347`:
    `gridTemplateColumns: "1.15fr 1fr"` без breakpoint — на мобильном
    разваливается (известный долг P2, подтверждён).
25. **OAuth refresh: пустой `refresh_token` принимается молча** —
    `core/services/oauth_store.py:232`: при malformed-ответе провайдера старый
    токен сохраняется без warning-лога.

## Проверено и проблем не найдено

- Пароли/сессии: `secrets.token_urlsafe(32)`, HMAC-хеши токенов, httpOnly cookies.
- CSRF middleware с осмысленными exempt для OAuth callbacks.
- CSP/HSTS/security headers, CORS без wildcard+credentials.
- `credentials: "include"` на всех fetch фронтенда; редиректы захардкожены.
- Alembic-цепочка целостна, `_JSONB`/`_UUID` декораторы корректны для SQLite-тестов.
- Stripe: подпись webhook проверяется; lockout/recovery-флоу в auth на месте.
- Нарушений правила «core/pipeline не импортируют adapters» не найдено.
- Захардкоженных секретов в репозитории не найдено.

## Отброшенные ложные срабатывания агентов-аудиторов
