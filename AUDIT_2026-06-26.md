## КРИТИЧЕСКИЕ блокеры (5) — чинить до любого запуска монетизации

### 1. Цены глубоко убыточны (Финансы)
Один поиск на 50 лидов стоит **~$1.7–2.0** реальных API-затрат (доминирует Google Place Details на reviews/Enterprise SKU: 50 × $0.028 = $1.40 + Haiku-скоринг).
- Pro $29/мес за 200 поисков = **~$370 себестоимости** → убыток ~12x.
- Team $79/мес за 500 поисков = **~$925 себестоимости**.
- Даже Free (5 поисков) = ~$9.25 затрат на $0 выручки.

**Что делать:** поднять цены 5–10x, ИЛИ резать глубину enrichment, ИЛИ убрать `reviews` из `DETAILS_FIELD_MASK` (переход с $0.028 на $0.020 SKU = −30% к доминирующей статье), ИЛИ перейти на модель лид-кредитов. Промоделировать worst-case (все планы выжаты) до включения биллинга.
`usage_tracker.py:54-61` vs `en.ts:814-826`

### 2. Нет cost-cap — расходы могут улететь незаметно (Финансы)
`usage_tracker.get_user_usage` / `total_cost_usd` считаются, но **нигде не потребляются** — ни в enforcement, ни в админке, ни в алертах. Один абьюзный/багнутый тенант крутит дневной лимит лидов (до 500/день на 'power', 25000 на 'unlimited') без денежного потолка. Узнаешь об этом из счёта Google.
**Что делать:** завести get_user_usage в (а) админку как лидерборд расходов по юзерам, (б) жёсткий месячный USD-потолок в `pipeline/search.py`, (в) Sentry/Slack-алерт при превышении дневного $-порога.
`usage_tracker.py:164-192`

### 3. Нет SEO-инфраструктуры (Маркетинг)
Нет `sitemap.ts`, нет `robots.ts`, нет OG/Twitter-картинок. Все публичные страницы — `"use client"`, поэтому **не могут экспортировать metadata вообще**. `/pricing`, `/vs/apollo|clay|lusha`, `/changelog`, `/help` — неиндексируемы и при шеринге дают пустую карточку. `/vs/*` — самый высоко-ROI органический актив категории — невидим для Google.
**Что делать:** перевести публичные страницы в server components (интерактив вынести в маленькие client-детей), добавить per-page metadata + sitemap.ts + robots.ts + OG-картинки (можно next/og для `/vs/*`).
`frontend/app/` (нет sitemap.ts, robots.ts)

### 4. Холодные письма без unsubscribe / List-Unsubscribe / почтового адреса (GDPR/CAN-SPAM)
Путь аутрича шлёт `<p>{body}</p>` без opt-out ссылки, без RFC 8058 List-Unsubscribe заголовка, без физического адреса. Единственный "unsubscribe" в коде — на внутреннем дайджесте владельцу, не на письмах скрейпленным третьим лицам. **Нарушает CAN-SPAM (US) и GDPR/ePrivacy (EU)** — юридический блокер для EU/US/UK.
**Что делать:** вшить обязательный unsubscribe-футер (ссылка + почтовый адрес) и List-Unsubscribe заголовок в `build_raw_message`, на основе per-tenant sender identity. Блокировать отправку без адреса.
`routes/gmail.py:354-377`, `integrations/gmail.py:186-218`

### 5. Нет recipient-level suppression / do-not-contact списка (GDPR/CAN-SPAM)
Получатель, который отписался, не может быть навсегда подавлен. Поле `blacklisted` — это search-side дедуп (скрыть лид из будущих поисков), а не глобальный suppression по email. Тот же бизнес-email, заскрейпленный в другом поиске, снова контактопригоден.
**Что делать:** tenant-scoped suppression-таблица по нормализованному email; проверять в send-пути и enrichment; авто-наполнять из кликов по unsubscribe и opt-out реплаев.
`db/models/lead.py:103-105`

## Сквозные темы (всплыли у нескольких агентов)

**A. Каша в тарифах — 3 несовместимые таксономии планов** (Финансы + Продукт + Маркетинг)
- Pricing-страница: free/pro/team
- In-app billing: free/personal_pro/team_standard/team_pro (причём team_standard и team_pro оба покупают один Stripe-price 'agency')
- Backend-капы: free/solo/agency/pro/power/unlimited
Планы 'solo' и 'power' недостижимы через Stripe. При включении биллинга — сломается. Свернуть в ОДИН канонический набор (free/pro/team), использовать те же строки в `PLAN_DAILY_LEAD_CAP`, `plan_for_price`, checkout. Добавить тест "каждый UI-план → реальный price + реальный cap".

**B. Цена Pro противоречит сама себе: $19 на лендинге vs $29 на /pricing** (Финансы + Маркетинг) — bait-and-switch в момент конверсии. Один источник правды.

**C. Русские артефакты в коде** (Backend + Интеграции + Архитектура + Продукт) — нарушают hard-констрейнт "English-first":
- `adzuna.py:110` — тег `'Нанимают маркетолога'` пишется в lead.tags, попадёт в CRM/экспорты US/UK/EU юзеров
- `website.py` — паттерн `vk`, телефонный regex на `+7/8`, кириллические хинты, `Accept-Language: ru`
- `pipeline/search.py` — хардкод русских строк прогресса (на web они мертвы, т.к. фронт строит свою локаль, но на Telegram-пути всплывают)

**D. SSRF redirect-bypass в website.py** (Backend + Безопасность — оба нашли независимо, HIGH) — `follow_redirects=True` после единственной pre-flight проверки. Лид-сайт может 30x-редиректнуть на `169.254.169.254` (cloud metadata) или RFC1918. URL приходят из результатов поиска → атакоуправляемы. На Railway = эксфильтрация метаданных.

**E. Документы лгут о коде** (Продукт + DevOps + Документация) — PLAN.md и PRODUCT_READINESS_AUDIT.md описывают до-пивотный Telegram/RU-продукт (2GIS, VK, Bitrix24, amoCRM, рубли, ЮKassa, 152-ФЗ). Любой (человек или агент), читающий их как актуальные, получает неверную картину. CLAUDE.md — единственный доверенный источник состояния.

## По направлениям: ключевое

### 1. Архитектура (82)
Правило отсутствия восходящих импортов реально соблюдается (grep подтвердил). Sinks-паттерн образцовый (2 адаптера: web_api + telegram_v2). Разбивка роутов зрелая (app.py = 411 строк, 30 модулей). **Минусы:** `pipeline/search.py` — 1251-строчный процедурный монолит (функция на 880 строк мешает 5 задач); `routes/_helpers.py` — 1063-строчная свалка из 25 ответственностей; `routes/leads.py` — 1488 строк бизнес-логики в адаптере.
*Quick win:* добавить guard-тест в CI (import-linter), фиксирующий правило слоёв, чтобы не регрессировало.

### 2. Backend (82)
Блокирующий I/O везде корректно вынесен в `to_thread`, все httpx-клиенты с `__aexit__` через `async with` — утечек нет. Конкурентность ограничена семафорами. **2 дефекта (HIGH), оба в website.py:** SSRF redirect-bypass (см. D) и `max_bytes` применяется ПОСЛЕ буферизации всего тела в память (`resp.text[:max_bytes]` — decompression-bomb DoS).
*Quick wins:* `follow_redirects=False` + ревалидация каждого хопа; `client.stream` с aiter_bytes; `Accept-Language: en`.

### 3. Frontend (78)
strict TS реально соблюдён: **0 any**, 0 console.log, 0 TODO/FIXME. API-слой отличный (типизированный `request<T>`, ApiError, AbortController). Гидрация корректна. **Минусы:** все 47 страниц — `"use client"`, RSC не используются; огромные файлы (leads/page.tsx — 1580 строк, 41 хук); accessibility тонкая (231 кнопка, 18 aria-label); leads-страница путает loading и error (упавшая загрузка = пустой CRM).
*Quick wins:* aria-label на icon-кнопки; reschedule поллинга в catch; убрать `"use client"` со статичных страниц.

### 4. БД (86) — лучшее
Цепочка из 55 миграций идеально линейна (root → head, без дыр и ветвлений). Каждый NOT NULL на заполненную таблицу с server_default. `_JSONB`/`_UUID` TypeDecorators корректны. FK-индексы покрыты (миграция 0049, 34 индекса). **Минусы (low/medium):** 0049.downgrade() дропает индексы, которые создали ранние миграции (ломает их downgrade); 2 FK без индекса замедляют CASCADE при удалении юзера (`client_reports.created_by_user_id`, `team_seen_leads.first_user_id`).

### 5. Безопасность (78)
Сильное намеренное усиление: argon2 + constant-time dummy-hash против энумерации, HMAC-токены в хранилище, подписанный+nonce OAuth state, реальный DNS-резолвящий SSRF-guard, HMAC-вебхуки, Fernet-шифрование токенов, lockout, audit log, строгий CSP. SQL полностью параметризован — инъекций нет. **Дефекты:** SSRF redirect-bypass (HIGH, см. D); password-reset/verify/invite токены хранятся **в plaintext** (HIGH — утечка БД = захват аккаунтов; `hash_token()` уже есть в коде); in-memory rate-limiter fails-open на нескольких репликах + обходится через спуфинг X-Forwarded-For (MEDIUM).
*Quick wins:* `secrets.compare_digest` в require_api_key; хешировать токены через существующий hash_token().

### 6. Интеграции (74)
Все OAuth-провайдеры единообразны, Stripe-вебхук верифицируется constant-time + идемпотентность, ключи нигде не захардкожены. Retry/timeout централизованы. Graceful degradation при отсутствии ключа. **Дефекты:** русские артефакты (HIGH, см. C); Foursquare/Yelp на устаревающих v3 API без fallback/телеметрии — при sunset молча отдадут пусто (MEDIUM); adzuna `_detect_country` дефолтит на 'gb' для неизвестных регионов → географически неверные лиды.

### 7. Тесты и CI (68)
495 тестов в 74 файлах, сильнейшее покрытие безопасности (IDOR-изоляция, Stripe-вебхуки, webhook HMAC/SSRF, lockout). Дисциплинированный мокинг. **Главные дыры:** core search pipeline (1251 строк) имеет только no-op smoke-тест (HIGH); **0 тестов фронта** на 159 ts/tsx (HIGH); **в CI нет никакого security-сканирования** — ни pip-audit/npm audit, ни SAST, ни secret-scan (HIGH) — при том что продукт держит OAuth-токены и Stripe-ключи; нет измерения coverage.
*Quick wins:* добавить pip-audit + npm audit + gitleaks в CI; тесты на `_passes_language_filter` (зафиксировать "no Russian market"); pytest-cov.

### 8. DevOps (62)
Есть забота: fail-fast на отсутствие prod-секретов, CORS '*' guard, security-заголовки, реальный CI против живого Postgres. **Дефекты:** Dockerfile делает `pip install .` по floating `>=`, **uv.lock не используется ни одним build-путём** — два билда одного SHA могут отгрузить разные версии (HIGH); Railway zero-downtime health-check **не подключён** (нет healthcheckPath в railway.json), хотя CLAUDE.md это утверждает; `/health/db` из доков вообще не существует в коде (HIGH); `python-multipart>=0.0.5` допускает CVE-2024-53981; Next.js закреплён на устаревшем 14.2.5; entrypoint.sh авто-stamp'ит alembic head при провале миграции — маскирует drift/data-loss.
*Quick wins:* `"healthcheckPath": "/health"` в railway.json; поднять python-multipart до >=0.0.18; next до >=14.2.30; пиннованный install в Dockerfile.

### 9. Дизайн/UX (76)
Зрелый, на бренде: дизайн-токены точно по DESIGN_BRIEF, "Convioo" везде (Leadgen только в .prototype-reference/dev-README/legacy-ключах). Ядро потока полное. Mobile-адаптив реальный. **Дефекты:** **ноль скелетон-лоадеров** хотя бриф их прямо требует — страницы мигают пусто→поп (HIGH); пункт навигации "New search" ведёт на Dashboard, а не на /app/search (HIGH); нативный `prompt()` для "Save search" выпадает из дизайн-системы; live-прогресс — поллинг 4с с частично синтетическим баром, не SSE.

### 13. Производительность (58)

*Quick wins:* переиспользовать единый Redis-пул; GZipMiddleware; сделать REDIS_URL обязательным в проде; --workers (после фикса broker'а).

### 14. Юр.соответствие (52)
